# Audit Flow.io du 20 juillet 2026

## Synthese

Le projet compile et son image SPIFFS est constructible. Les protections Web
ajoutees lors du precedent audit (mot de passe aleatoire, authentification,
CSRF, controle d'origine et en-tetes de securite) sont bien presentes.

La version reste adaptee a un banc ou a un reseau d'administration isole, mais
pas encore a une mise en production sur un reseau non maitrise. Les mises a jour
reseau non signees sont bien bloquees par defaut. Il manque toutefois une chaine
de mise a jour signee et exploitable, et les televersements locaux ne verifient
pas encore de signature. Le transport d'ecran distant HMI UDP a ete supprime.

## Validation executee

- Compilation `Waveshare-ESP32-S3` : succes.
- RAM statique : 122 436 / 327 680 octets (37,4 %).
- Flash applicative : 1 619 093 / 2 097 152 octets (77,2 %).
- Construction SPIFFS : succes.
- `scripts/verify_release.py` : `release verification: OK`.
- Recherche des API memoire C dangereuses dans `src/` et `include/` : aucune
  utilisation de `strcpy`, `strcat`, `sprintf`, `vsprintf`, `gets` ou `scanf`.
- Tests materiels : non executes (materiel non connecte dans cet audit).
- Le test Unity natif est configure. Son execution locale reste impossible sur
  ce poste tant que `gcc`/`g++` ne sont pas installes; la compilation ESP32-S3,
  qui utilise sa propre chaine Xtensa, reussit.

## Constats prioritaires

### P1 - Chaine de mise a jour signee absente

`FLOW_ALLOW_UNSIGNED_UPDATES` vaut zero par defaut, ce qui bloque correctement
les mises a jour reseau actuelles. Si cette protection est levee, le module
accepte `http://`, utilise HTTP par defaut et transmet les flux directement a
`Update.write` ou au programmateur FlowIO. Les televersements locaux restent
possibles sans verification de signature. Le manifeste contient des SHA-256,
mais n'est pas signe.

Impact : les mises a jour reseau sont indisponibles en configuration sure. Si
elles sont reactivees telles quelles, un serveur compromis, une alteration DNS
ou un attaquant sur le chemin peut fournir une image arbitraire. Un compte Web
administrateur compromis peut aussi televerser une image locale non signee.

Actions : imposer HTTPS avec validation de certificat, verifier le SHA-256 avant
activation, signer le manifeste ou l'image avec une cle publique embarquee,
refuser les versions inferieures et tester le rollback. Appliquer aussi la
verification aux televersements locaux.

### P2 - Administration Web sans confidentialite de transport

Le serveur est expose en HTTP. L'authentification Digest evite d'envoyer le mot
de passe en clair, mais ne chiffre ni les donnees d'administration, ni les
reponses, ni les commandes. Le jeton CSRF est egalement observable sur le lien.

Actions : terminer TLS sur une passerelle locale de confiance ou ajouter HTTPS
sur l'equipement ; conserver un VLAN d'administration et ne jamais publier le
port 80 sur Internet.

### Corrige - Limitation et journalisation des echecs Web

Le middleware suit 32 adresses sources. Il limite chaque source a cinq echecs
Digest par minute puis la bloque cinq minutes. Les entrees encore bloquees ne
sont pas evincees par de nouvelles sources. Un plafond global de 40 echecs par
minute bloque aussi les attaques distribuees pendant une minute. Les reponses
utilisent `429` avec `Retry-After`; le challenge Digest initial n'est pas compte.

### P2 - Socle ESP32 en fin de maintenance

La compilation utilise Arduino-ESP32 2.0.17 (`framework-arduinoespressif32`
3.20017) et une chaine GCC 8.4. Les bibliotheques applicatives sensibles ont ete
mises a jour, mais le socle reste ancien.

Actions : preparer une branche de migration vers Arduino-ESP32 3.x/ESP-IDF 5.x,
avec tests reseau, Ethernet, MQTT TLS, OTA, PSRAM, SPIFFS et temporisations.

### Partiellement corrige - Tests et integration continue

Un environnement PlatformIO `native` et un workflow CI couvrent maintenant les
cinq tests de calcul, la compilation, SPIFFS, la verification de livraison et un
seuil de 85 % de la partition firmware. Les chemins Web, MQTT, OTA, HMI,
persistance et securites de relais ne sont toutefois pas encore couverts.

Actions : ajouter un environnement `native` pour la logique pure, une CI qui
compile firmware et SPIFFS puis lance `verify_release.py`, et des tests sur cible
pour les interfaces materiel/reseau.

### Corrige - Garde-fou de partition applicative

La partition applicative est occupee a 77,2 %, soit environ 478 Ko de marge.
Ce n'est pas un depassement, mais TLS, verification de signatures et futures
fonctionnalites reduiront cette marge.

La CI echoue maintenant au-dessus de 85 %. Il reste a suivre le tas interne, la
fragmentation et les pics de pile sur materiel reel.

### P3 - Industrialisation incomplete

Aucune licence, aucun SBOM et aucun workflow CI n'ont ete trouves. Le manifeste
est integre mais non signe. Le build genere par defaut un identifiant horodate,
donc deux constructions ne sont identiques que si `FLOW_BUILD_REF` est fixe.

Actions : ajouter une licence explicite, un SBOM SPDX/CycloneDX, documenter les
licences tierces, signer les livraisons et fixer l'identifiant de build en CI.

## Points positifs confirmes

- MQTT refuse explicitement `mqtt://` et configure le bundle de certificats pour
  `mqtts://`.
- Les mots de passe Web et du point d'acces de provisioning sont generes avec
  `esp_random()` lorsqu'aucune valeur de fabrication n'est fournie.
- Les routes Web mutantes passent par authentification et validation CSRF ; les
  WebSockets exigent une origine correspondant a l'hote.
- Les dependances directes sont versionnees et le mode de compatibilite est
  strict.
- Les tailles, erreurs de flux et ecritures OTA partielles sont controlees.
- Les partitions A/B OTA sont presentes et la partition inactive est selectionnee.

## Validation materielle encore requise

Avant production, tester sur l'appareil reel : demarrage et recuperation des
acces, Wi-Fi/Ethernet, MQTT TLS et certificats, relais et interverrouillages,
Qwiic/I2C/DS2484/ADS1115, Nextion UART, persistance NVS, coupures pendant
OTA, selection/rollback des partitions, watchdogs, piles des taches, tas interne
et PSRAM apres fonctionnement prolonge.

## Ordre recommande

1. Securiser et signer toute la chaine OTA.
2. Ajouter la confidentialite de transport Web.
3. Mettre en place CI, tests et seuils de taille/memoire.
4. Migrer le socle Arduino-ESP32 sur une branche dediee.
5. Realiser la campagne materielle et l'industrialisation (SBOM, licence, cles).
